扫描项目中的所有 .vue, .ts, .js 文件，找出超过文件大小限制的文件，并提供重构建议。

## 文件大小限制

| 文件类型 | 最大行数 | 建议拆分点 | 紧急阈值 |
|---------|---------|-----------|---------|
| Vue 组件 | 300 行 | 250 行 | 500 行 |
| TypeScript | 400 行 | 350 行 | 600 行 |
| JavaScript | 400 行 | 350 行 | 600 行 |
| Store/State | 300 行 | 250 行 | 500 行 |
| Rust 文件 | 300 行 | 250 行 | 500 行 |

## 拆分原则

1. 超过 250 行的组件应该考虑拆分
2. 超过 300 行的组件必须拆分
3. 优先提取业务逻辑到 composables
4. 按功能模块拆分 UI 组件
5. 复用现有组件，避免重复代码

## TypeScript/Rust 分工规则

### 优先 Rust 的场景

- 文件系统操作
- 日志记录
- 搜索/过滤/排序等计算密集型操作
- 敏感数据处理（API Key 等）

### 保持 TypeScript 的场景

- AI API 调用（HTTPS 易调试）
- Vue 组件和 Composables（UI 绑定逻辑）
- 业务状态管理（Store）
- 业务逻辑和 UI 交互

### 迁移检查清单

新增功能时自检:

1. [ ] 是否涉及文件 I/O？ -> Rust
2. [ ] 是否是计算密集型？ -> Rust
3. [ ] 是否涉及敏感数据？ -> Rust
4. [ ] 纯 UI 绑定/业务逻辑？ -> TypeScript

## 审计步骤

1. 使用 Glob使用 Glob 工具扫描所有 Vue 和 TypeScript 文件
2. 使用 Read 工具读取每个文件并计算行数
3. 识别超过限制的文件
4. 分析文件结构，提供具体的重构建议

## 重构建议模板

对于每个违规文件，提供以下信息：

### 文件概况
- 文件路径
- 当前行数
- 超出限制的行数
- 紧急程度标记（🔥 > 500行, ⚠️ > 350行）

### 重构建议

**Vue 组件拆分策略：**

如果组件超过 300 行，建议：

1. **提取 composables**
   - 查找超过 30 行的函数，提取到 `composables/use{功能名}.ts`
   - 例如：`useScriptMetadata`, `useCommandFilter`

2. **拆分子组件**
   - 将模板部分按功能模块拆分
   - 每个子组件保持在 150-250 行
   - 命名规范：父组件名+功能名，如 `ScriptEditorDialog` → `ScriptEditorActions`

3. **提取通用组件**
   - 识别重复的 UI 模式，提取到可复用组件

**TypeScript 文件拆分策略：**

如果 TypeScript 文件超过 400 行，建议：

1. **按领域拆分类型**
   - 将大型 schema.ts 按功能领域拆分
   - 例如：`types/command.ts`, `types/script.ts`

2. **提取工具函数**
   - 将独立的工具函数提取到单独文件

3. **模块化导出**
   - 使用 `index.ts` 统一导出

### 优先级评估

- **🔥 紧急（>500行）**：立即拆分，严重影响维护性和 AI 效率
- **⚠️ 警告（350-500行）**：计划拆分，避免进一步膨胀
- **ℹ️ 提示（接近限制）**：关注增量，避免超出

## 输出格式

最后生成一份汇总报告，包含：

### 📊 审计汇总
- 扫描文件总数
- 违规文件总数
- 按严重程度分类统计

### 📋 违规文件列表（按严重程度排序）

🔥 **紧急（立即处理）**
1. 文件名 - 行数 - 建议拆分方案

⚠️ **警告（计划处理）**
2. 文件名 - 行数 - 建议拆分方案

### 💡 重构建议总结
- 总计需要拆分的文件数
- 建议的拆分策略总结
- 预计完成后 AI 效率提升
